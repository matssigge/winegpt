## Toolchain & workflow

Everything runs through Docker Compose via `just`. There is no host-side `cargo`/`npm` workflow ‚Äî the host only needs `docker`, `just`, and (optionally) `mise` to install them.

- `just install` ‚Äî build the dev images
- `just dev` ‚Äî run backend + frontend together (compose project `wine`)
- `just backend-run` / `just frontend-dev` ‚Äî run one service
- `just backend-test` ‚Äî backend tests in the isolated `wine-test` stack
- `just frontend-test` ‚Äî runs `scripts/smoke-test.mjs` (a static structural check, not real tests)
- `just frontend-e2e-test` ‚Äî Playwright in the isolated stack, tears down on exit
- `just test` ‚Äî backend + frontend smoke + e2e
- `just down` ‚Äî stop the dev stack

Three Compose files, each with a distinct project name and purpose:
- `compose.yml` (`-p wine`) ‚Äî local dev with persistent `postgres-data` volume, Postgres exposed on `127.0.0.1:6432`
- `compose.test.yml` (`-p wine-test`) ‚Äî backend tests + Playwright; Postgres on tmpfs, never reuses dev data
- `compose.prod.yml` ‚Äî runtime stack

Service ports: backend `:3000`, frontend dev `:5273`, frontend preview `:4173`.

## Running a single test

Backend tests run serially because they share one Postgres database that `test_support::test_database()` truncates between tests. To run one test, pass the filter through compose:

```
docker compose -p wine-test -f compose.test.yml run --rm backend-test \
  cargo test --manifest-path backend/Cargo.toml <pattern> -- --test-threads=1
```

Keep `--test-threads=1`. Parallel test runs will race on the shared schema.

A single Playwright spec:

```
docker compose -p wine-test -f compose.test.yml run --rm frontend-e2e \
  npx playwright test e2e/auth.spec.js
```

## Backend architecture (`backend/`)

Axum + SQLx (Postgres) + Tokio. Layered as the standards require:

- `main.rs` ‚Äî Axum router, CORS, error ‚Üí HTTP-status mapping. Handlers stay thin: authenticate, call domain module, map errors.
- `lib.rs` ‚Äî config parsing (`BACKEND_BIND_ADDR`, `DATABASE_URL`, `FRONTEND_ORIGINS`), health response. Pure functions, unit-tested.
- `db.rs` ‚Äî pool connect + `sqlx::migrate!("./migrations")` runs on startup; failures here are fatal.
- `auth.rs`, `collections.rs`, `wines.rs`, `entries.rs` ‚Äî domain modules owning their SQL and typed error enums (`AuthError`, `CollectionError`, `CollectionWineError`, `EntryError`). Handlers map these to `(StatusCode, ErrorResponse)` in `main.rs`.
- `test_support.rs` ‚Äî `test_database()` truncates all tables between tests; gated behind a `TEST_MUTEX` so test ordering is deterministic.

Migrations live in `backend/migrations/` and are embedded into the binary at build time by the `sqlx::migrate!` macro ‚Äî schema changes require a rebuild, not just a restart. Auth uses bearer tokens in an `Authorization: Bearer <token>` header backed by a `sessions` table; password hashing is Argon2.

## Frontend architecture (`frontend/`)

ReScript + React 19 + Vite + Tailwind v4. ReScript is configured `in-source` (`rescript.json`), so each `Foo.res` produces a sibling `Foo.bs.js`. **Both files are committed.** When editing, change the `.res` file ‚Äî the `.bs.js` is regenerated by `rescript build`.

Per the standards: write new frontend code in ReScript. Use JS only for thin interop (browser APIs awkward in ReScript, third-party bindings, build/runtime glue). Never have one ReScript module import another through its generated `.bs.js`.

Module layout under `src/`:
- `Main.res` mounts `<App />`. `App.res` is the top-level state container (auth, collections, wines, entries, composer modes) ‚Äî most features hang off here as `useState` slices and effects.
- `AppShell.res` ‚Äî authenticated layout that fans state and callbacks out to feature components.
- `api/` ‚Äî `ApiClient.res` (fetch wrapper, base URL detection, bearer headers, error decoding) plus per-domain API modules (`AuthApi`, `CollectionApi`, ‚Ä¶).
- `auth/`, `collections/`, `wines/`, `entries/`, `ui/` ‚Äî feature folders, typically split into `Model` (types/decoders), `State` (reducers + API calls), and presentational components.
- Session token persists in `localStorage` via `SessionStorage`/`SessionBootstrap`; the selected collection id persists via `CollectionSelectionStorage`.

API base URL resolution (`ApiClient.res`): use `VITE_API_BASE_URL` if set; otherwise derive from `window.location` (with a `frontend ‚Üí backend` host swap so the dev container talks to the backend container).

## Code conventions worth remembering

- Tailwind for styling; use `clsx` for conditional classes ‚Äî do not concatenate class strings.
- Prefer explicit SQL in domain modules; do not embed SQL in HTTP handlers.
- Errors: typed enums in domain modules, mapped to user-safe codes at the HTTP boundary. Don't leak internal details to clients.
- Don't use `--no-verify` or skip the `wine-test` stack when "fixing" tests ‚Äî the isolated stack is the test environment.
